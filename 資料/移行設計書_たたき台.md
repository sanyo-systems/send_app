# 移行設計書 たたき台

## 1. 影響範囲

- 本資料は、ワークスペース `C:\apiとapp用` を前提にした移行方針の整理である。
- 調査対象は `Setting.ini`、および `back_end` / `front_end` 以外の既存 Python ファイルに限定する。
- 本資料は設計整理のみを目的とし、実装、サーバー起動、データベース接続、CSV監視起動、外部通信、データ更新は行わない。
- 既存の `back_end` / `front_end` 配下のファイルは、本資料では移行元資産として扱わず、存在を前提にしない。

## 2. 目的

- 既存の Windows デスクトップ主体の Python システムを、FastAPI を用いたバックエンドと React + TypeScript を用いたフロントエンドへ段階的に移行する。
- 既存業務ロジックを可能な限り再利用しつつ、UI と外部I/Oを分離し、保守しやすい構成へ整理する。
- `Setting.ini` を起点とした既存設定資産を尊重しながら、将来的に `.env` とアプリ内設定へ整理可能な形に再構成する。

## 3. 現行システムの概要

### 3.1 主な責務

- `main.py`
  - アプリ起動入口
  - スレッド起動
  - UI 起動
  - CSV監視、再送、ヘルス監視、アップデート確認の統合制御

- `communication`
  - `config_loader.py`: `Setting.ini` 読み込み、設定解釈
  - `recorder_client.py`: 記録計へのソケット送信
  - `plc_sender.py`: PLC向け送信
  - `send_queue.py`: 送信キュー、永続化、再送管理

- `csv_monitor`
  - `csv_watcher.py`: CSV監視、CSV解析、送信予約生成、送信トリガー判定
  - `retry_worker.py`: 再送処理

- `database`
  - `check_history.py`: Access から履歴参照
  - `teitei.py`: Access 履歴参照、チェック履歴登録の一部
  - `access_writer.py`: Access への履歴書き込み
  - `path_resolver.py`: Access パス補正

- `history`
  - 送信履歴、予約状態、失敗キューなどのローカル状態管理

- `ui`
  - `gamegame.py`: Tkinter 画面全体
  - `validation.py`: 入力チェック
  - `coment.py`: コメント保存と読込

- `utils`
  - ファイルロック、CSV移動、キー正規化、状態保存などの共通処理

- `monitoring`
  - ログ設定、スレッド監視、ハートビート

### 3.2 現行構成の特徴

- UI、設定、業務ロジック、外部I/Oが密結合している。
- Access DB、UNC共有、記録計ソケット通信、PLC連携など外部依存が多い。
- `Setting.ini` に外部接続先と業務設定が混在している。
- Tkinter UI の中に業務ロジックと画面制御が同居している。

## 4. 移行方針の基本方針

### 4.1 基本方針

- 既存コードをそのまま Web 用に書き換えるのではなく、責務ごとに分離して再配置する。
- FastAPI は「薄い API 層」とし、既存ロジックを service 層から呼び出す構成を目指す。
- React は Tkinter UI の移植ではなく、ユーザー操作単位で画面と状態管理を再構成する。
- 外部I/Oは adapter 層に隔離し、業務ロジックから直接触れない構造にする。
- `Setting.ini` は初期段階では互換レイヤーとして扱い、段階的に `.env` とアプリ設定へ整理する。

### 4.2 非機能面の方針

- 既存システムへの影響を避けるため、段階移行を前提とする。
- 初期段階では自動送信や常駐監視よりも、参照・登録などの明示操作機能を優先する。
- 外部接続先を直接変更しない構成を優先し、検証用設定と本番用設定を明確に分離する。

## 5. 推奨アーキテクチャ

### 5.1 バックエンド構成案

### 5.2 フロントエンド構成案

- `hooks`
  - API呼び出しと画面状態管理
- `api`
  - Axios 設定と API 呼び出しラッパー
- `types`

## 6. 移行対象の分類

### 6.1 再利用優先

- `communication/config_loader.py`
  - `Setting.ini` の既存解釈ルールを流用しやすい。
- `csv_monitor/csv_watcher.py`
  - 時刻正規化、CSV解析、送信予定生成ロジックを切り出して再利用しやすい。
- `database/check_history.py`
  - 履歴参照ロジックとして再利用しやすい。
- `database/teitei.py`
  - Access 履歴取得ロジックの再利用候補。
- `database/access_writer.py`
  - Access 書き込みロジックの再利用候補。
- `history/*`
  - 予約状態や送信状態の保存ルールは再利用価値が高い。
- `utils/*`
  - キー正規化、ファイルロック、状態保存などは共通部品化しやすい。

### 6.2 切り出して再構成

- `main.py`
  - 起動制御は Web 化では再設計が必要。
- `communication/send_queue.py`
  - キュー管理は流用可能だが、API / ワーカー構成に合わせて分離が必要。
- `communication/recorder_client.py`
  - 通信処理自体は再利用候補だが、Web API 直下ではなく adapter 化が必要。
- `communication/plc_sender.py`
  - 同上。
- `csv_monitor/retry_worker.py`
  - 常駐ワーカーとして分離設計が必要。

### 6.3 作り直し前提

- `ui/gamegame.py`
  - Tkinter 前提のため、React に合わせて画面単位で再設計する。
- `ui/validation.py`
  - 一部ルールは流用可能だが、入力UI依存を切り離して再構成した方がよい。
- `update/update_service.py`
  - デスクトップ更新機構のため、Web構成では別検討とする。

## 7. 設定移行方針

### 7.1 `Setting.ini` の分類

#### `.env` 候補

- `DEFAULT_SHARE_DIR`
- `ACCESS_FILE`
- `ACCESS_FILE_2`
- `CSV_FOLDER1`
- `CSV_FOLDER2`
- `RECORDER_IP_ADRESS*`
- `RECORDER_PORT*`
- `WAIT_TIME1`
- `PLC_MODE`
- `ACT_LOGICAL_STATION_NUMBER`

#### アプリ設定候補

- `UI_REC_TYPE`
- `RECORDER_TYPE*`
- `CSV_FILE*`
- `RECORDER_GROUP*`
- `RECORDER_GROUP_NAME*`
- 炉名とCSV名の対応ルール
- 業務上の対象炉構成

### 7.2 移行方針

- 初期段階では `.env` と `Setting.ini` を併用する。
- バックエンドで `.env` を先に読み、未指定項目のみ `Setting.ini` から補完する互換レイヤーを用意する。
- 連番設定は将来的に配列形式の設定ファイル、またはDBマスタへの移行を検討する。

## 9. バックエンド API の初期案

### 9.1 初期フェーズで優先する API

- `GET /health`
  - 稼働確認

- `GET /settings/ui-context`
  - UI表示に必要な炉種、炉一覧、表示ラベルなどを返す

- `GET /furnaces`
  - 対象炉一覧取得

- `GET /history/1h`
  - 1H履歴取得

- `GET /history/4h`
  - 4H履歴取得

- `GET /history/latest`
  - 最新履歴取得

- `POST /checks/1h`
  - 1H登録

- `POST /checks/4h`
  - 4H登録

- `GET /comments`
  - コメント取得

- `POST /comments`
  - コメント登録

- `GET /reservations`
  - 送信予約一覧

- `DELETE /reservations/{reservation_id}`
  - 送信予約取消

- `GET /comm-logs`
  - 通信ログ表示用データ取得

### 9.2 後続フェーズの API

- `POST /csv/parse`
  - CSV内容の確認用解析

- `POST /send/preview`
  - 送信予定内容の確認

- `POST /send/execute`
  - 手動送信

- `POST /monitoring/start`
  - 常駐監視の開始

- `POST /monitoring/stop`
  - 常駐監視の停止

注記:
常駐監視や自動送信系は、初期フェーズでは対象外とし、後続フェーズで別ワーカー化を前提に設計する。

## 12. リスクと注意点

### 12.1 技術リスク

- Access DB 依存が強く、将来的な置換を見据えた repository 分離が必要。
- `Setting.ini` の連番定義は Web 化後の保守性が低い。
- Tkinter UI 内部に埋め込まれた業務ロジックの切り出しに工数がかかる。
- CSV監視と送信キューは API サーバー本体から分離しないと運用が重くなりやすい。

### 12.2 運用リスク

- 現行の UNC パス、記録計IP、PLC連携は環境差分の影響を受けやすい。
- 試験用設定と本番設定の混在があるため、設定の明確な分離が必要。
- 自動送信機能は初期段階から組み込むと既存業務への影響確認が難しくなる。
