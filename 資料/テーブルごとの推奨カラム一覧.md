# テーブルごとの推奨カラム一覧

## 1. 影響範囲

- 本資料は `C:\apiとapp用` ワークスペース内の既存コード、`Book1.xlsx`、および参照資料をもとにした調査整理である。
- 既存テーブルを流用する前提で、**新規で作成する必要のあるテーブル**のみに絞って整理する。
- 実装、DB作成、既存データ更新、サーバー起動、外部接続は行っていない。

## 2. 前提

### 2.1 流用する既存テーブル

今回の整理では、以下の既存資料テーブルを流用する前提とする。

- `users`
  - 社員系の代替として流用
- `equipment`
  - 炉名系の代替として流用
- `equipment_groups`
  - 炉種系の代替として流用
- `inspection_records`
  - 点検ヘッダ系の代替候補として流用可能
- `inspection_results`
  - 点検明細系の代替候補として流用可能

### 2.2 今回は新規作成しないもの

以下は既存テーブル流用または設計簡略化により、今回新規作成対象から外す。

- `factory`
- `ro_type`
- `furnace`
- `employee`
- `reservation_record`
- `send_queue_record`
- `send_result_record`
- `check_type_master`

### 2.3 今回新規で必要なもの

既存テーブルだけでは不足するため、新規で必要になるのは次のテーブルである。

- `factory`
- `recorder_ip`
- `furnace_recorder_map`
- `check_record`
- `csv_send_record`
- `comment_record`

## 3. 新規で作成する必要のあるテーブル

### 3.1 `factory`

用途:
工場マスタ。

必要な理由:

- `users.factory` は文字列保持であり、業務テーブルからの参照先としては扱いにくい。
- 今回の新規テーブル側で `factory_name` 文字列を持ち回るより、`factory_id` に統一した方が整合性を保ちやすい。

推奨カラム:

| カラム名 | 型の例 | 必須 | キー | 説明 |
| --- | --- | --- | --- | --- |
| `factory_id` | `BIGINT` | 必須 | PK | 工場ID |
| `factory_name` | `VARCHAR(50)` | 必須 | UNIQUE候補 | 工場名 |
| `display_order` | `INT` | 任意 |  | 表示順 |
| `is_active` | `BOOLEAN` | 必須 |  | 使用中フラグ |
| `created_at` | `DATETIME` | 必須 |  | 作成日時 |
| `updated_at` | `DATETIME` | 必須 |  | 更新日時 |

### 3.2 `recorder_ip`

用途:
記録計IPマスタ。

必要な理由:

- 記録計IPは業務上の識別要素であり、文字列を各テーブルへ重複保持するよりマスタ化した方がよい。
- `check_record` や `csv_send_record` から同じIP定義を参照できるようにするため。

推奨カラム:

| カラム名 | 型の例 | 必須 | キー | 説明 |
| --- | --- | --- | --- | --- |
| `recorder_ip_id` | `BIGINT` | 必須 | PK | 記録計IP ID |
| `ip_address` | `VARCHAR(45)` | 必須 | UNIQUE | IPアドレス |
| `port_no` | `INT` | 必須 |  | ポート番号 |
| `group_no` | `INT` | 任意 |  | 既定グループ番号 |
| `is_active` | `BOOLEAN` | 必須 |  | 使用中フラグ |
| `created_at` | `DATETIME` | 必須 |  | 作成日時 |
| `updated_at` | `DATETIME` | 必須 |  | 更新日時 |

### 3.3 `furnace_recorder_map`

用途:
炉と記録計の対応表。  
`Setting.ini` の `RECORDER_IP_ADRESS*`、`RECORDER_PORT*`、`RECORDER_GROUP*`、`CSV_FILE*` の連番構造を、Web/API 向けに正規化して保持する。

必要な理由:

- 既存流用候補の `equipment`、`equipment_groups`、`users` には、記録計IPとの対応情報が存在しない。
- 今回の業務では「どの炉がどの記録計IP・ポート・グループ番号に対応するか」が必要になる。
- この対応は、今回の資料群の既存テーブルには見当たらないため新規作成が必要。

推奨カラム:

| カラム名 | 型の例 | 必須 | キー | 説明 |
| --- | --- | --- | --- | --- |
| `furnace_recorder_map_id` | `BIGINT` | 必須 | PK | 対応ID |
| `equipment_id` | `UUID` | 必須 | FK | `equipment.id`。炉に相当 |
| `equipment_group_id` | `UUID` | 任意 | FK | `equipment_groups.id`。炉種補助 |
| `factory_id` | `BIGINT` | 必須 | FK | `factory.factory_id` |
| `recorder_ip_id` | `BIGINT` | 必須 | FK | `recorder_ip.recorder_ip_id` |
| `csv_file_name` | `VARCHAR(100)` | 任意 |  | 対応CSVファイル名 |
| `is_active` | `BOOLEAN` | 必須 |  | 使用中フラグ |
| `created_at` | `DATETIME` | 必須 |  | 作成日時 |
| `updated_at` | `DATETIME` | 必須 |  | 更新日時 |

補足:

- `watch_folder_type` は今回の用途では必須ではないため外してよい。
- `recorder_ip` は文字列ではなく `recorder_ip_id` 参照に寄せる方が一貫する。
- `group_no` は `recorder_ip` テーブル側で管理する前提とする。
- 同一IPでグループ違いが必要な場合は、`recorder_ip` に別レコードを作成して表現する。

### 3.4 `check_record`

用途:
今回の 1H/4H チェック履歴を、既存 inspection 系とは別にシンプルに保持するためのテーブル。

必要な理由:

- 既存の `inspection_records` / `inspection_results` は汎用点検システム向けで、今回の 1H/4H チェック専用構造としてはやや重い。
- 今回は `1H` / `4H` 固定であり、既存業務の表示形式や入力値に合わせたシンプルな履歴が必要。
- 既存コードでも、1H/4H チェックは Access 側で専用履歴として扱われているため、新規テーブルを分けた方が整理しやすい。

推奨カラム:

| カラム名 | 型の例 | 必須 | キー | 説明 |
| --- | --- | --- | --- | --- |
| `check_record_id` | `BIGINT` | 必須 | PK | チェック履歴ID |
| `equipment_id` | `UUID` | 必須 | FK | `equipment.id`。炉に相当 |
| `equipment_group_id` | `UUID` | 任意 | FK | `equipment_groups.id`。炉種補助 |
| `factory_id` | `BIGINT` | 必須 | FK | `factory.factory_id` |
| `employee_id` | `UUID` | 任意 | FK | `users.id`。入力者 |
| `recorder_ip_id` | `BIGINT` | 任意 | FK | `recorder_ip.recorder_ip_id` |
| `record_time` | `DATETIME` | 必須 | INDEX候補 | 登録日時 |
| `check_hour` | `TINYINT` | 必須 |  | 何時台のチェックか |
| `check_type` | `VARCHAR(10)` | 必須 |  | `1H` / `4H` |
| `display_text` | `TEXT` | 必須 |  | 入力値を画面用にまとめた文字列 |
| `raw_value_json` | `TEXT` | 任意 |  | 元の入力値をJSON保存 |
| `created_at` | `DATETIME` | 必須 |  | 作成日時 |

補足:

- `record_time` があれば `record_date` は必須ではない。
- `employee_code`、`employee_name` は `employee_id` から `users` を参照して取得する運用なら省略可能。
- `recorder_ip` は文字列ではなく `recorder_ip_id` 参照で持つ方がよい。
- `check_type_master` は不要で、`check_type` を `1H` / `4H` 固定運用で十分。
- `display_text` は、画面にそのまま出したい要約文字列を保存する列である。
  例:
  - ピット炉なら `ACT 123`
  - 焼準系なら `729/689/175`
  元入力を毎回組み立て直さずに一覧表示したい場合に使う。不要なら `raw_value_json` のみで運用する選択もある。

### 3.5 `csv_send_record`

用途:
CSV由来の送信対象兼送信履歴を保持するテーブル。  
送信予約、未送信、送信済み、失敗、取消を `send_status` に集約して扱う前提。

必要な理由:

- 既存資料群には、CSV送信専用の業務テーブルが存在しない。
- 今回の業務では `start_time`、`end_time`、送信内容、送信状態が重要であり、inspection 系テーブルでは代替しにくい。
- `reservation_record`、`send_queue_record`、`send_result_record` を別で作らず、ここに集約する方針の中心となる。

推奨カラム:

| カラム名 | 型の例 | 必須 | キー | 説明 |
| --- | --- | --- | --- | --- |
| `csv_send_record_id` | `BIGINT` | 必須 | PK | CSV送信レコードID |
| `equipment_id` | `UUID` | 必須 | FK | `equipment.id`。炉に相当 |
| `equipment_group_id` | `UUID` | 任意 | FK | `equipment_groups.id`。炉種補助 |
| `factory_id` | `BIGINT` | 必須 | FK | `factory.factory_id` |
| `recorder_ip_id` | `BIGINT` | 任意 | FK | `recorder_ip.recorder_ip_id` |
| `instruction_no` | `VARCHAR(100)` | 必須 |  | 指示Noまたは送信識別子 |
| `instruction_text` | `TEXT` | 任意 |  | 実際の送信内容 |
| `trigger_time` | `DATETIME` | 任意 | INDEX候補 | 実際に送信判定に使う時刻 |
| `send_status` | `VARCHAR(20)` | 必須 | INDEX候補 | `PENDING` / `READY` / `SENT` / `FAILED` / `CANCELLED` |
| `retry_count` | `INT` | 任意 |  | 再送回数 |
| `sent_at` | `DATETIME` | 任意 |  | 送信完了日時 |
| `last_attempt_at` | `DATETIME` | 任意 |  | 最終送信試行日時 |
| `error_message` | `TEXT` | 任意 |  | エラー内容 |
| `created_at` | `DATETIME` | 必須 |  | 作成日時 |
| `updated_at` | `DATETIME` | 必須 |  | 更新日時 |

補足:

- `send_status` に集約する前提なら、未送信や送信済みの判定はこのテーブルだけで可能。
- `reservation_record` を作らない代わりに、送信予約状態は `PENDING` / `READY` で表現する。
- `send_queue_record` を作らない代わりに、キュー状態は `send_status` と `retry_count` で管理する。
- `send_result_record` を作らない代わりに、最終結果は `send_status`、`sent_at`、`error_message` で保持する。
- `start_time` と `end_time` は送信時刻算出の中間情報であり、DBへ残す主目的がなければ省略可能。
- `source_file_name` と `source_file_path` は、炉が特定できてファイル監査要件もないなら省略可能。
- 送信試行履歴を全件残したくなった時点で、別テーブル追加を再検討すればよい。

### 3.6 `comment_record`

用途:
炉ごとのコメント履歴。現行の `comment/*.json` 相当。

必要な理由:

- 既存資料の `abnormal_board_comments` は異常ボード用であり、今回の炉コメント履歴の完全代替にはならない。
- 今回のコメントは炉単位の運用コメントであり、異常管理とは分けて持つ方が自然。

推奨カラム:

| カラム名 | 型の例 | 必須 | キー | 説明 |
| --- | --- | --- | --- | --- |
| `comment_record_id` | `BIGINT` | 必須 | PK | コメントID |
| `equipment_id` | `UUID` | 必須 | FK | `equipment.id`。炉に相当 |
| `equipment_group_id` | `UUID` | 任意 | FK | `equipment_groups.id`。炉種補助 |
| `factory_id` | `BIGINT` | 必須 | FK | `factory.factory_id` |
| `comment_text` | `TEXT` | 必須 |  | コメント本文 |
| `created_at` | `DATETIME` | 必須 | INDEX候補 | 登録日時 |

補足:

- `comment_time` は不要と判断してよい。`created_at` がその役割を兼ねる。
- 入力者情報を画面や監査で使わない前提なら、`employee_id` と `employee_name` も不要でよい。
- 過去時刻を手入力する要件が出た場合のみ、別の業務時刻列追加を検討すればよい。

## 6. 補足整理

### 6.1 既存流用との対応

- 炉名
  - `equipment` を流用
- 炉種
  - `equipment_groups` を流用
- 社員
  - `users` を流用
- 工場
  - 既存 `users.factory` は文字列のため、新規 `factory` を作成して参照を寄せる

### 6.2 `csv_send_record` に集約する考え方

- `send_status` によって未送信、送信待ち、送信済み、失敗、取消を判断する
- `start_time`、`end_time`、`trigger_time` によって送信タイミングを判断する
- `retry_count`、`error_message`、`sent_at` によって送信結果の追跡を行う

この前提なら、別の予約・キュー・送信結果テーブルは不要。

### 6.3 将来見直しの可能性がある点

- 工場を文字列ではなくマスタ化したくなった場合
  - `factory_name` を `factory_id` に置き換える
- 送信試行履歴を全件残したくなった場合
  - `send_result_record` を追加検討する
- 予約と送信結果を厳密に分けたくなった場合
  - `reservation_record` を追加検討する
